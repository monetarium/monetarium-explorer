## Требования к feature: Таблица страницы Blocks — поддержка монет SKA{n}

### Контекст

Страница `/blocks` с пагинацией использует унаследованный от Decred макет таблицы, в котором отображается единственная колонка со значением `DCR` без учёта типов монет. Таблица Latest Blocks на главной странице уже обновлена для поддержки двухмонетной модели (строки VAR + SKA{n}, раскрывающийся аккордеон). Данная функция приводит таблицу страницы Blocks в соответствие с этой моделью, сохраняя при этом колонки, уникальные для страницы Blocks: `Version` и `Time (UTC)`.

---

### 1. Структура колонок

Обновлённая таблица Blocks должна содержать следующие колонки в указанном порядке:

| Колонка      | Подпись (десктоп) | Подпись (мобильный/сокращённая) |
| ------------ | ----------------- | ------------------------------- |
| Height       | Height            | Height                          |
| Transactions | Transactions      | Txn                             |
| Votes        | Votes             | -                               |
| Tickets      | Tickets           | -                               |
| Revocations  | Revocations       | -                               |
| VAR          | VAR               | VAR                             |
| SKA          | SKA               | SKA                             |
| Size         | Size              | Size                            |
| Version      | Version           | -                               |
| Age          | Age               | Age                             |
| Time (UTC)   | Time (UTC)        | Time (UTC)                      |

Колонка `DCR` заменяется колонками `VAR` и `SKA`.

---

### 2. Поведение раскрывающихся строк

Каждая строка блока должна быть раскрываемой — идентично поведению таблицы Latest Blocks:

- **Состояние по умолчанию:** свёрнуто.
- **По клику:** строка раскрывается и показывает:
  - одну под-строку VAR
  - множество под-строк SKA{n} (**все эмитированные SKA{n}**, даже если в блоке нет транзакций или значения равны 0)
- В подстроках значения колонки Transactins **не должны** включать Vote, Ticket и Revocation транзакции.
- Под-строки используют те же колонки, что и родительская строка. Отдельные заголовки для под-строк не предусмотрены.
- В под-строках не отображаются значения в колонках Votes, Tickets, Revocations, Version, Age и Time — в этих ячейках показывается `—`.

Данные под-строк по типу монеты:

**Под-строка VAR:**

- Transactions: количество обычных (Regular) VAR-транзакций в блоке
- VAR: общее количество монет VAR в блоке
- Size: размер VAR-транзакций в байтах

**Под-строки SKA{n} (SKA1, SKA2, …):**

- Transactions: количество обычных (Regular) SKA{n}-транзакций в блоке
- SKA: общее количество монет SKA{n} в блоке
  - Если значение равно **0**, отображается **0**, а не `—`
- Size: размер SKA{n}-транзакций в байтах

---

### 3. Колонка Transactions — родительская строка

Количество транзакций в колонке `Transactions` родительской строки должно исключать типы Vote, Ticket и Revocation:

- Transactions = все транзакции за вычетом Vote, Tickets, Revokes

Это соответствует поведению таблицы Latest Blocks.

---

### 4. Значения VAR и SKA — точность отображения

Суммы отображаются по правилу трёх цифр:

- Используются **только три значащие цифры**
- Применяются буквенные сокращения: `K`, `M`, `B`, `T`

Примеры:

- `4.67K`
- `2.94M`

Правила отображения отсутствующих значений:

- Для числовых значений:
  - используется **0**, если значение равно нулю
- Прочерк (`—`) используется **только если значение неприменимо для данного типа строки**  
  (например, SKA{n}-значения для строки VAR)

---

### 5. Шеврон / элемент раскрытия

На каждой строке должен присутствовать индикатор-шеврон (аналогично Latest Blocks), сигнализирующий о возможности раскрытия:

- В свернутом состоянии — указывает на возможность раскрытия
- При раскрытии — меняет направление

---

### 6. Колонки Version и Time (UTC)

Эти колонки уникальны для страницы Blocks и должны быть сохранены:

- `Version` отображается только в родительской строке  
  В под-строках — `—`
- `Time (UTC)` отображается только в родительской строке  
  В под-строках — `—`
- `Age`:
  - относительное время
  - рендерится через JS (`data-time-target="age"`), аналогично Latest Blocks

---

### 7. Адаптивное поведение

Следовать той же логике контрольных точек (breakpoints), что и:

- таблица Latest Blocks
- существующая таблица Blocks

Требования:

- mobile-first подход
- корректная работа в тёмной теме
- сокращённые подписи колонок на мобильных устройствах

---

### 8. Вне области действия

- Логика пагинации не изменяется
- Порядок сортировки не изменяется (по Height, по убыванию)
- Ссылки:
  - `/side` (orphaned blocks)
  - `/disapproved` (PoS invalidated blocks)  
    остаются без изменений

---

### 9. Критерии приёмки

- [ ] Каждая строка блока раскрывается и показывает под-строки VAR + **все эмитированные SKA{n}**
- [ ] Колонка `DCR` заменена колонками `VAR` и `SKA`
- [ ] Количество транзакций в родительской строке исключает Vote, Ticket и Revocation
- [ ] Количество транзакций в под-строке VAR не включает Vote, Ticket и Revocation
- [ ] Количество транзакций в под-строках SKA{n} не включает Vote, Ticket и Revocation
- [ ] Значения **0 отображаются как `0`, а не `—`**
- [ ] Прочерк используется только для неприменимых значений
- [ ] Все SKA{n} отображаются при раскрытии, даже если в блоке нет транзакций
- [ ] Под-строки SKA{n} корректно отображают данные (Transactions, SKA, Size)
- [ ] Колонки `Version` и `Time (UTC)` присутствуют и не затронуты механизмом раскрытия
- [ ] Брейкпоинты адаптивности и тёмная тема сохранены
- [ ] Отображение сумм соответствует правилу трёх цифр (K/M/B/T)
