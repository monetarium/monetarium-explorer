# Visual Blocks — требования из созвона с заказчиком

## §0. Контекст

Этот документ содержит **только требования из записи созвона** (`call.txt`).
Статус: **черновик для мержа** в основной spec [`spec.md`](spec.md).

Конфликты с `spec.md` явно помечены тегом `⚡ Конфликт с spec.md` — разрешать их здесь **не нужно**, это задача шага мержа.

---

## §1. Заголовок плитки — размер блока вместо суммы в DCR

**Текущее поведение:**
Центральная часть заголовка показывает общую сумму блока с единицей «DCR» (например, «486 DCR»).

**Требуемое поведение:**

- Заменить «486 DCR» на **размер блока** (в байтах / КБ).
- Если помещается в заголовок — дополнительно показывать **% от максимального размера блока**.
- Если не помещается — минимум: показать только размер блока.
- Максимальный размер блока берётся из chain spec и передаётся сервером.

Остальные элементы заголовка не изменяются:

- кликабельная высота блока (ссылка `/block/{height}`) / «Mempool» (ссылка `/mempool`);
- возраст блока (справа).

**Данные:** `Block.Size` (uint32, байты) — `db/dbtypes/types.go`; `BlockBasic.FormattedBytes` (string) — `explorer/types/explorertypes.go`.

> ⚡ **Конфликт с spec.md §2.2:** `spec.md` требует заменить «DCR» на вертикальный список
> сумм по монетам (VAR + SKA{n}). Созвон требует показывать **размер блока**.
> Нужно разрешить при мерже.

## §2. Строка наград (Rewards row) — удалить

**Текущее поведение:**
ROW 1 (`block-rewards`) — зелёная полоса, показывает пропорцию между PoW-наградой майнеру,
PoS-наградой стейкерам, Project Fund и комиссиями (соответствующие сегменты разной яркости зелёного).

**Требуемое поведение:**
Строку наград **удалить полностью**.

**Причина:** Полоса показывает соотношение PoW-награды и PoS-награды. В Monetarium оно
всегда фиксировано (50/50), поэтому не несёт информации. После удаления строки голосов
(ROW 2 / синяя) идёт сразу за заголовком.

> ⚡ **Конфликт с spec.md §3:** `spec.md` сохраняет Rewards row — только удаляет
> сегмент «Project Fund» и делает PoW мультимонетным. Созвон удаляет **всю строку**.
> Нужно разрешить при мерже.

## §3. Строка голосов (Votes row) — три состояния по результату голосования

**Текущее поведение:**
5 слотов; голос «за» = синий, голос «против» = красный, пустой слот = нет цвета.

**Требуемое поведение:**
На каждый блок случайно выбираются ровно **5 билетов**. Кошелёк-владелец может:
ответить «да», ответить «нет», или оказаться офлайн и не проголосовать вовсе.

Три состояния элемента голоса:

| Статус                           | Цвет        | Описание                                                                  |
| -------------------------------- | ----------- | ------------------------------------------------------------------------- |
| Voted YES — проголосовал «за»    | **Синий**   | Согласен с блоком; возвращается стоимость билета + награда за голос       |
| Voted NO — проголосовал «против» | **Красный** | Не согласен с блоком (редко: сбой или атака)                              |
| Did not vote — не проголосовал   | **Серый**   | Кошелёк был офлайн; возвращается **только** стоимость билета, награды нет |

Дополнительные правила:

- «Пустые слоты» (для выравнивания до 5, если слотов нужно меньше) — оставить прозрачными (текущее поведение).
- Существующие **всплывающие подсказки** для голосов — оставить без изменений.

> **Примечание:** Текущий код различает два состояния через `voteValid` (true/false).
> Третье состояние «did not vote» потребует дополнительного поля в данных голоса.

## §4. Строка билетов (Tickets row) — без изменений

Жёлтые купленные билеты + оранжевые ревокации + прозрачные пустые слоты — **без изменений**.
Существующие **всплывающие подсказки** — оставить без изменений.

## §5. Строка транзакций → три полосы индикатора блока

**Текущее поведение:**
ROW 4 (`block-transactions`) — серая полоса; каждая обычная транзакция — отдельный серый
(`#c4cbd2`) сегмент, ширина пропорциональна сумме транзакции.

**Требуемое поведение:**
Серую строку транзакций **удалить** и заменить **полосами трёх типов** из компонента
`indicator-fill` (тот же компонент, что используется на главной странице):
TOTAL, VAR и SKA{n} — по одной на каждую активную SKA-монету.

### 5.1. Плитка мемпула

Идентично индикатору на главной странице:

- **TOTAL bar** — общее заполнение мемпула vs. максимальный размер блока.
- **VAR bar** — заполнение VAR-транзакциями vs. VAR-квота (10% от max block size).
- **SKA{n} bar** — одна полоса на каждую активную SKA-монету vs. её квота (90% / кол-во активных SKA).

### 5.2. Плитка намайненного блока

Те же три полосы, но **по фактическим данным блока**:

- **TOTAL bar** — фактический размер блока / максимальный размер блока.
- **VAR bar** — суммарный размер VAR-транзакций блока / VAR-квота.
- **SKA{n} bar** — суммарный размер SKA{n}-транзакций блока / SKA{n}-квота.

### 5.3. Алгоритм квот (как на главной)

- VAR quota = 10% от maxBlockSize.
- SKA pool = 90% от maxBlockSize, делится поровну между всеми активными SKA-монетами.

> ⚡ **Конфликт с spec.md §6:** `spec.md` сохраняет строку транзакций и адаптирует её
> к мультимонетной модели. Созвон **полностью заменяет** её индикатором.
> Нужно разрешить при мерже.

## §6. Всплывающие подсказки для полос индикатора

Для компонента `indicator-fill` на `/visualblocks` подсказки показывают
**количество транзакций**, а не процент занятого места (как на главной странице).

Визуальный компонент (`indicator-fill`) один и тот же для обоих типов плиток.
Различается только знаменатель в тексте подсказки — в силу разной природы данных.

### §6.1. Плитка мемпула — «% от мемпула»

Мемпул — живой срез в реальном времени: состав и размер известны прямо сейчас.
Знаменатель доступен без каких-либо исторических данных.

| Полоса | Содержимое подсказки                         |
| ------ | -------------------------------------------- |
| TOTAL  | «N транзакций всего — X% от мемпула»         |
| VAR    | «N VAR-транзакций — X% от VAR-мемпула»       |
| SKA{n} | «N SKA{n}-транзакций — X% от SKA{n}-мемпула» |

Данные источника: `TrimmedMempoolInfo.CoinFills` + `CoinStats` — уже вычислены
сервером и передаются по WebSocket (`getmempooltrimmed`). Изменений в серверной части не требуется.

### §6.2. Плитки намайненных блоков — «% от ёмкости блока»

Для намайненного блока мемпул на момент майнинга **не восстановим**: БД не хранит
snapshot мемпула при приёме блока. При приходе нового блока (`BlockHandler` →
`CollectAndStore` в `mempool/monitor.go`) состояние мемпула обновляется только в
памяти (три реализации `MempoolDataSaver`: `DataCache`, `explorerUI`, `PubSubHub`) —
ни одна не пишет в БД. Единственная «pool»-таблица — `stats` (`db/dcrpg/internal/stats.go`) —
хранит **ticket pool**, а не мемпул транзакций.

Вместо «% от мемпула» используется **«% от ёмкости блока»**:

| Полоса | Содержимое подсказки                       |
| ------ | ------------------------------------------ |
| TOTAL  | «N транзакций всего — X% от ёмкости блока» |
| VAR    | «N VAR-транзакций — X% от VAR-квоты»       |
| SKA{n} | «N SKA{n}-транзакций — X% от SKA{n}-квоты» |

**Обоснование выбора знаменателя:**
«% от ёмкости блока» точнее соответствует тому, что видит пользователь: полоса
заполнена на X% — подсказка показывает те же X% от максимума. Знаменатель стабилен
и понятен без исторического контекста. Значение `maxBlockSize` уже передаётся
сервером (см. §1).

Рассматривались два варианта реализации для плиток блока:

#### Вариант 1 — Историческое хранение snapshot мемпула

Добавить новую таблицу `mempool_snapshots`, куда при каждом `BlockHandler` записывается
состояние мемпула на момент приёма блока.

**Что нужно:**

- Новая таблица в БД (схема и миграция), включая per-coin breakdown.
- Новая реализация `MempoolDataSaver`, пишущая в БД.
- Синхронизация порядка вызовов: `BlockHandler` и `StoreBlock()` работают конкурентно
  в одной группе — нужно согласовать `block_id` между ними.
- Обработка реорганизаций цепи: при реорганизации нода откатывает блок (orphan) — запись
  snapshot в БД становится «висячей». Нужно либо настроить `ON DELETE CASCADE` на
  внешнем ключе `block_id`, либо явно удалять записи в Go-коде при обработке отката.
- Ретроспективных данных нет: snapshot доступен только для блоков начиная с момента развёртывания.

**Оценка:** 2–4 дня. Основная сложность — синхронизация порядка вызовов и обработка реорганизаций,
а не сами данные (в `MempoolInfo.CoinStats` всё необходимое уже есть).

---

#### Вариант 2 — Всплывающая подсказка на основе ёмкости блока ✅ Выбран для v1

Знаменатель — `maxBlockSize`, который уже передаётся сервером (см. §1).
Изменения только в клиентской части. Никаких изменений в БД, серверной части или
конвейере сбора данных.

**Оценка:** 2–4 часа, только клиентская часть.

---

#### Сравнение вариантов для плиток блока

| Критерий         | Вариант 1 (исторический snapshot)      | Вариант 2 (ёмкость блока) |
| ---------------- | -------------------------------------- | ------------------------- |
| Трудозатраты     | 2–4 дня                                | 2–4 часа                  |
| Риск             | Средний (синхронизация, реорганизации) | Минимальный               |
| Ретроспектива    | Нет (только новые блоки)               | Есть для всех блоков      |
| Точность метрики | «N tx = X% мемпула»                    | «N tx = X% ёмкости блока» |
| Изменения в БД   | Да (новая таблица, миграция)           | Нет                       |

**Решение для v1:** реализуется **Вариант 2**.
Вариант 1 может быть реализован в следующей итерации при необходимости.
