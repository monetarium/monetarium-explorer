# Visual Blocks — поддержка мультимонетной модели Monetarium (requirements)

## 0. Контекст

Страница `/visualblocks` отображает **последние блоки и текущий мемпул в виде «плиток»**, где элементы блока (награды, голоса, билеты, обычные транзакции) визуализированы горизонтальными цветными строками. Раздел унаследован от Decred-кодовой базы и в текущем виде поддерживает только одну монету (DCR), модель Project Fund (Decred Treasury) и одну шкалу субсидии на блок.

В Monetarium действуют другие инварианты (см. [CLAUDE.md](../../../CLAUDE.md), [core/product.md](../../core/product.md), [core/pages.md](../../core/pages.md)):

- Поддерживается несколько монет: **VAR** и до 255 типов **SKA{n}** (SKA1, SKA2, …).
- **Один блок может содержать транзакции нескольких монет**, однако **одна транзакция всегда односоставна по монете**.
- **Комиссия транзакции уплачивается в монете самой транзакции** (VAR-tx → fee в VAR, SKA{n}-tx → fee в SKA{n}).
- **Treasury (Project Fund) отсутствует** — соответствующая шкала в Decred-схеме на Monetarium неприменима.
- Staking-механика остаётся **VAR-only** (голосование, билеты, ревокации существуют только в VAR; PoS-награда выплачивается в VAR — см. [homepage-metrics/spec.md §4.1](../homepage-metrics/spec.md)).
- PoW-награда **существует в каждой эмитированной монете блока**: VAR-награда за блок плюс SKA{n}-награды по числу эмитированных типов SKA (см. [homepage-metrics/spec.md §4.2](../homepage-metrics/spec.md)).
- **Точность чисел:** VAR — 8 знаков (помещается в `float64`); SKA — 18 знаков (**не помещается в `float64`** — атомы должны идти строкой/`big.Int` до самого DOM, без преобразования в `float`).

Данный документ описывает, что нужно **создать / обновить / удалить** на странице `/visualblocks`, чтобы привести её в соответствие с мультимонетной моделью Monetarium.

## 1. Структура страницы (общий вид)

Структура страницы сохраняется:

- слева — плитка **Mempool** (текущее состояние мемпула);
- справа от неё — последовательность плиток **последних блоков** по убыванию высоты.

Сколько плиток блоков пользователь видит — **зависит от размера окна**: страница подстраивается под доступную ширину и высоту и показывает столько плиток, сколько помещается без обрезания и переноса; то же пересчитывается при изменении размера окна. На широком мониторе плиток видно больше, на узком — меньше; самая свежая всегда левее самой старой. При приходе нового блока самая свежая плитка появляется слева сразу после плитки мемпула, а самая старая из видимых уходит с экрана.

Это поведение **сохраняется как есть** — мультимонетные правки его не затрагивают.

> Для разработчиков: текущая реализация держит небольшой DOM-кэш плиток (фиксированный потолок на стороне сервера) и переключает видимость каждой плитки CSS-классом на клиенте (пересчёт при загрузке и на `resize`). При мультимонетных правках важно не поменять эти инварианты случайно — не увеличивать/уменьшать серверный потолок, не ломать пересчёт видимости и не оставлять «осиротевшие» плитки в DOM при приходе нового блока.

Внутри каждой плитки сохраняются четыре строки:

1. **Rewards row** — субсидия + комиссии (изменяется, см. §3).
2. **Votes row** — голосования (VAR-only, см. §4).
3. **Tickets row** — билеты + ревокации (VAR-only, см. §5).
4. **Regular transactions row** — обычные транзакции, мультимонетные (см. §6).

**Терминология** (используется по всему документу):

- **Плитка** (`block tile` / `mempool tile`) — большой элемент сетки: одна за мемпул, до 30 за последние блоки. Содержит заголовок и четыре строки.
- **Строка** (`row`) — одна из четырёх горизонтальных строк внутри плитки (соответствует `block-rewards` / `block-votes` / `block-tickets` / `block-transactions`).
- **Сегмент** — отдельный элемент внутри строки: цветной кусок Rewards row (PoW / PoS / Fees / Fund), отдельный голос / билет / ревокация или отдельная транзакция в Regular transactions row.

> Шаблон и динамический рендер по WebSocket должны выдавать **структурно идентичный DOM** (классы, `data-*`-хуки, формат `title`). Это критично из-за наличия двух пайплайнов рендеринга (HTTP-шаблон и JS-контроллер), которые мутируют один и тот же DOM — см. [code-analysis/visualblocks/patterns.md](../../code-analysis/visualblocks/patterns.md).

## 2. Заголовок плитки (Block info / Mempool info)

### 2.1. Кликабельный заголовок плитки

Без изменений:

- для плитки мемпула — «Mempool» с ссылкой `/mempool`;
- для плитки блока — высота блока с ссылкой `/block/{height}`.

### 2.2. Сумма блока — замена единой строки `DCR`

**Текущее поведение:**
Отображается одна запись вида `486 DCR` (целое число + единица `DCR`). Значение — `block.TotalSent` в Decred.

**Требуемое поведение:**

- Жёстко зашитая подпись `DCR` **должна быть удалена** во всех местах:
  - шаблон [`cmd/dcrdata/views/visualblocks.tmpl`](../../../cmd/dcrdata/views/visualblocks.tmpl) (строки с `<span class="unit"> DCR</span>` для плитки мемпула и каждой плитки блока);
  - JS-контроллер [`cmd/dcrdata/public/js/controllers/visualBlocks_controller.js`](../../../cmd/dcrdata/public/js/controllers/visualBlocks_controller.js) (`makeMempoolBlock`, `_makeBlockNode`).
- Вместо одного значения отображается **вертикальный список сумм по монетам**, аналогично таблицам Latest Blocks / Block Details:
  - первая запись — `VAR` (всегда, даже если 0);
  - далее записи `SKA1`, `SKA2`, … — **только** для тех SKA{n}, у которых в данном блоке/мемпуле есть транзакции с ненулевой суммой;
  - порядок: VAR, затем SKA{n} по возрастанию `n`.
- Формат значений — **правило трёх значащих цифр** с суффиксами `K / M / B / T` (как на главной), чтобы поместиться в плитку.
- Подпись монеты (`VAR`, `SKA1`, …) ставится **справа** от значения, отдельной шапки `Coin / Amount` нет.

**Точность отображения (правило применяется только к видимым пользователю числам):**

- Заголовок плитки — это **отображаемое числовое значение**, его пользователь читает. Поэтому здесь правило точности из [core/constraints.md](../../core/constraints.md) применяется в полной мере:
  - VAR — до 8 знаков после запятой;
  - SKA{n} — до 18 знаков после запятой, **без** преобразования в обычное «дробное» число (так как 18 значащих знаков не помещаются в стандартный машинный формат с плавающей точкой и неизбежно округляются при таком преобразовании, искажая последние знаки).
- На главной странице и в данной плитке отображение всё равно сжимается до **трёх значащих цифр + суффикс K/M/B/T** — но это сжатие должно идти **от точного исходного значения**, иначе у крупных SKA-сумм могут «съехать» уже первые три цифры (см. объяснение в §3.5 «почему точность нужна именно для чисел, а не для сегментов»).

### 2.3. Возраст блока (Age)

Без изменений: относительное время через `data-time-target="age"`.

## 3. Rewards row (строка наград)

### 3.1. Удаление сегмента «Project Fund»

`Project Fund` (сегмент `<span class="fund">`) — это Decred Treasury. В Monetarium казначейства нет (`/treasury` и `/treasurytable` возвращают 410 — см. [core/pages.md](../../core/pages.md)).

**Изменение:** сегмент `fund` должен быть **полностью удалён**:

- из шаблона [`visualblocks.tmpl`](../../../cmd/dcrdata/views/visualblocks.tmpl) — обе позиции (плитка мемпула и плитка блока, использующие `.Subsidy.Dev` и `.Subsidy.Developer` соответственно);
- из JS (`makeMempoolBlock`, `_makeBlockNode`) — обе ветки чтения `subsidy.developer || subsidy.dev`;
- сопутствующее CSS-правило `.fund` в [`public/scss/visualblocks.scss`](../../../cmd/dcrdata/public/scss/visualblocks.scss) допускается удалить или оставить «осиротевшим»; визуально оно нигде не должно проявляться.

Поскольку Project Fund отсутствует как сущность, бэкенд-поля `BlockSubsidy.Dev` и `chainjson.GetBlockSubsidyResult.Developer` со стороны Monetarium-node всегда равны `0`. На странице эта величина больше **не отрисовывается** и **не учитывается** в подсчёте долей.

### 3.2. PoW reward — мультимонетная разбивка

**Текущее поведение:**
Один сегмент `<span class="pow">`, ширина = `flex-grow: {{toFloat64Amount .Subsidy.PoW}}`. Считает только VAR-награду (Decred-модель).

**Требуемое поведение:**

- Сегмент PoW-награды должен отражать **сумму PoW-вознаграждений по всем монетам блока**: VAR PoW + каждая эмитированная SKA{n} PoW.
- Визуально это **один сегмент `pow`**, представляющий совокупный «вес» PoW-награды блока. Всплывающая подсказка показывает разбивку по монетам:  
  `{"object": "PoW Reward", "totals": {"VAR": "<сумма>", "SKA1": "<сумма>", ...}}`.
- Для блоков, в которых выпуск SKA{n} ещё не состоялся, в `totals` отображается только VAR.
- **Точность чисел в подсказке** подчиняется общему правилу (VAR — 8 знаков, SKA — 18 знаков, без преобразования SKA в формат с плавающей точкой). **Ширина самого сегмента** — это визуальная пропорция и точности не требует, см. §3.5.

**Объяснение (зачем нужна разбивка по монетам).** В Decred за каждый блок майнеру начисляется одна сумма награды — в одной монете (DCR). В Monetarium ситуация принципиально иная: каждая эмитированная монета имеет **собственную субсидию за блок**. Например, в блоке, помимо VAR-награды майнеру, может быть начислена SKA1-награда и SKA2-награда, если эти монеты уже эмитированы. Пользователю при просмотре блока на этой странице важно увидеть, что блок «накормил майнера» сразу несколькими монетами — и какими именно. Сам сегмент PoW остаётся одним (он отражает «насколько блок был насыщен PoW-наградой»), а конкретные суммы по монетам пользователь видит во всплывающей подсказке. Мультимонетная природа блока при этом всё равно остаётся видна без наведения — в заголовке плитки (§2.2), где суммы по VAR и каждому эмитированному SKA{n} показаны отдельным вертикальным списком.

**Почему не подсегменты по монетам внутри сегмента.** Альтернатива «разделить `pow` на подсегменты — по одному на каждую эмитированную монету» была рассмотрена и отвергнута. Причины: (1) §3.5 прямо запрещает кросс-монетное визуальное сравнение по шкале — VAR-атомы (10⁸) и SKA-атомы (10¹⁸) нельзя класть на одну линейку; подсегменты разных монет рядом друг с другом провоцируют именно такое сравнение. (2) Сегмент комиссий (§3.4) выбран как «один сегмент + разбивка в подсказке» — если PoW сделать подсегментами, визуальный язык соседних сегментов Rewards row разойдётся. (3) Двойной пайплайн рендера (см. §1, §8) проще держать в синхронизации, когда DOM не содержит N-зависимого цикла подсегментов.

### 3.3. PoS reward

PoS-вознаграждение в Monetarium **VAR-only** (стейкинг — VAR). Сегмент `pos` остаётся без структурных изменений, но:

- всплывающая подсказка должна явно содержать монету: `{"object": "PoS Reward", "coin": "VAR", "total": "..."}`;
- размерность совместима с PoW-сегментом (см. §3.5).

**Объяснение (почему PoS остаётся VAR-only).** Стейкинговая часть протокола в Monetarium работает только на VAR: билеты покупаются в VAR, голосование за блок ведётся через VAR-билеты, награда голосующему выплачивается в VAR. SKA{n}-монет это не касается. Поэтому сегмент PoS — всегда «одна монета», и здесь мультимонетная разбивка не нужна. Указание монеты в подсказке делается ради единообразия со всеми другими сегментами страницы.

### 3.4. Transaction Fees — мультимонетная сумма

**Текущее поведение:**
Сегмент комиссий учитывает только сумму VAR-комиссий блока — других монет на Decred нет.

**Требуемое поведение:**

- Сегмент `fees` остаётся **единым** (не разбивается на подсегменты по монетам) с цветом, отличным от PoW/PoS.
- В Monetarium комиссия каждой транзакции платится в её собственной монете. На странице это отражается **визуально как один общий сегмент**, но всплывающая подсказка обязана содержать **разбивку по монетам**:  
  `{"object": "Tx Fees", "totals": {"VAR": "<сумма>", "SKA1": "<сумма>", ...}}`.
- Монеты с нулевой комиссией в `totals` **не включаются**.
- Ширина самого сегмента — визуальная пропорция, точности к ней не предъявляется (см. §3.5). Числа в подсказке отображаются по точным правилам (VAR — 8 знаков, SKA — 18 знаков).

**Объяснение (почему комиссии бывают в разных монетах).** В Monetarium действует строгое правило: транзакция «живёт» в одной монете и платит комиссию в ней же — VAR-транзакция платит комиссию в VAR, SKA1-транзакция платит в SKA1, SKA2 — в SKA2. Поэтому у блока, в котором были транзакции разных монет, существуют **раздельные счётчики комиссий** для каждой монеты — это не одна общая «копилка», а несколько параллельных. Сегмент на странице по-прежнему один (он отражает «насколько блок был насыщен комиссиями»), но конкретные суммы пользователь видит в подсказке — отдельно по каждой монете, у которой комиссии были.

### 3.5. Ширина цветных сегментов — визуальная подсказка, а не точное число

Сегменты Rewards row — это **визуальный индикатор пропорций** между PoW-наградой, PoS-наградой и комиссиями. Пользователь смотрит на них, чтобы понять «больше или меньше», а не «насколько именно в восьмом знаке после запятой».

Поэтому **точность 8/18 знаков к ширине сегментов не применяется**:

- Достаточно приблизительной оценки величины — глазом разница между «сегмент занял 47% строки» и «сегмент занял 47,0000001%» неразличима.
- Любой разумный способ округления при расчёте ширины допустим — даже если он стоил бы нескольких знаков после запятой.

**Важное ограничение шкалы — сравнивать можно только в рамках одной монеты.** VAR и SKA{n} — разные единицы измерения (как «литры» и «килограммы»): нельзя положить «50 VAR» и «50 SKA1» на одни весы и сказать, что они равны. Поэтому:

- PoW-VAR в сегменте сравнивается с другими VAR-величинами (например, с типовой VAR-субсидией за блок);
- PoW-SKA1 — с типовой SKA1-субсидией за блок;
- VAR-комиссии — с типовым VAR-объёмом комиссий;
- SKA1-комиссии — с типовым SKA1-объёмом комиссий, и т.д.

Каждый сегмент выражает **долю «своей» величины от типового максимума этой же монеты** — это даёт сопоставимые между собой пропорции (все в диапазоне «0..полностью заполнено») без склеивания разных единиц.

**Объяснение «зачем эта оговорка».** Если этого не сделать, получится визуальный казус: SKA-награда измеряется в атомах с 18 нулями, а VAR — в атомах с 8 нулями. Если сложить эти числа «в лоб», SKA-сегмент всегда выглядит в десятки миллиардов раз больше — даже когда фактически SKA-награда скромная. Поэтому сегменты вычисляются «каждый в своих координатах», а уже визуально пользователь видит, что у блока, например, «PoW почти полный, PoS невелик, комиссии минимальны» — без зависимости от того, в какой монете эти награды считаются.

**Точность сохраняется только там, где пользователь видит сами числа — но форма «видимости» у подсказки и у заголовка плитки разная:**

- **Всплывающие подсказки (§3.2, §3.4, §6.3)** — пользователь читает значение целиком, поэтому оно отображается с полной точностью: VAR — до 8 знаков после запятой, SKA — до 18 знаков без потери точности (без преобразования через формат с плавающей точкой).
- **Заголовок плитки (§2.2)** — пользователь видит сжатое представление: три значащие цифры + суффикс `K / M / B / T`. Само сжатие к точности не предъявляет требований, но **исходное значение для него должно быть точным** (VAR — 8 знаков, SKA — 18 знаков без потери точности). Иначе при крупных SKA-суммах потеря последних значащих знаков на этапе подготовки данных «сдвигает» уже первые три цифры результата.

В обоих случаях правило одно: до момента отображения число едет в форме, сохраняющей все знаки (для SKA — строкой/`big.Int`). Различается только то, что показывается пользователю в финале — полное число или его «трёхзначный» конспект.

## 4. Votes row (строка голосов)

Голосование в Monetarium — VAR-only, структура голосов наследуется из Decred-схемы.

**Изменений по логике нет:**

- по-прежнему до 5 голосов на блок;
- цвет по `voteValid`;
- ссылка на `/tx/<hash>`.

**Изменения по разметке:**

- из `title`-tooltip нужно убрать любые скрытые упоминания DCR (если есть);
- структура `title` остаётся `{"object": "Vote", "voteValid": "<true|false>"}`.

Пустые «слоты голосов» (если в блоке проголосовало меньше 5 билетов) сохраняются. Константа `5` остаётся захардкоженной как параметр стейкинга.

## 5. Tickets row (строка билетов и ревокаций)

Билеты и ревокации в Monetarium — VAR-only.

**Изменений по логике нет**, кроме:

- в `title`-tooltip билета явно добавить `"coin": "VAR"`:  
  `{"object": "Ticket", "coin": "VAR", "total": "<VAR-amount>", "vout": "...", "vin": "..."}`;
- аналогично для ревокаций;
- `total` отображается в VAR с 8 знаками точности (через `float64` допустимо).

Пустые слоты (до 20 на плитку: 20 - votes - tickets - revocations) сохраняются как захардкоженный stake-параметр.

## 6. Regular transactions row (строка обычных транзакций)

Обычные транзакции — **главное место, где плитка должна стать мультимонетной**, поскольку:

- в одном блоке могут присутствовать обычные транзакции в разных монетах (VAR-tx, SKA1-tx, SKA2-tx);
- каждая отдельная транзакция остаётся односоставной по монете (инвариант Monetarium).

### 6.1. Состав сегментов

В строке отображаются **все обычные (regular) транзакции блока**, без коинбейса. Каждая транзакция = один сегмент `<span class="block-tx">` с ссылкой на `/tx/<hash>`.

Не должно быть фильтрации SKA-транзакций или объединения их в один общий сегмент — каждая SKA-tx отображается как отдельный сегмент наравне с VAR-tx.

### 6.2. Размер сегмента транзакции

**Текущее поведение:**
Ширина сегмента транзакции пропорциональна VAR-сумме перевода (других монет в Decred нет, поэтому проблем нет).

**Требуемое поведение:**

- Ширина сегмента — **визуальная пропорция**, точности она не требует (так же, как для сегментов Rewards row, см. §3.5).
- Каждый сегмент получает ширину **пропорционально сумме своей транзакции в её собственной монете**:
  - VAR-tx сравнивается шириной с другими VAR-tx в этой же строке;
  - SKA1-tx — с другими SKA1-tx;
  - SKA2-tx — с другими SKA2-tx, и т.д.
- Сравнение «VAR vs SKA1» по абсолютному значению **не имеет смысла** (разные единицы) и не должно делать один сегмент огромным только из-за того, что его монета имеет больше знаков после запятой.

**Объяснение (зачем это нужно).** Если просто положить сумму SKA-перевода и сумму VAR-перевода на «одну линейку», SKA-сегмент зрительно «съест» всю строку — потому что у SKA внутреннее представление числа имеет на 10 нулей больше, чем у VAR. Пользователь увидит, что строка целиком закрашена SKA-цветом, и не поймёт, что в блоке вообще были VAR-транзакции. Поэтому сегменты разных монет считаются по своим «линейкам» — внутри одной монеты они корректно сопоставимы, между монетами визуально не конкурируют по абсолютной величине.

> Это устраняет [«тихий риск»](../../code-analysis/visualblocks/impact.md), при котором SKA-сегмент из-за разницы в шкале (атомов 10⁸ у VAR против 10¹⁸ у SKA) выглядел как «огромная» транзакция и заполнял собой всю строку, хотя его доля в собственной монете могла быть скромной.

### 6.3. Всплывающая подсказка транзакции

Всплывающая подсказка сегмента транзакции должна явно содержать монету и точную сумму перевода:

```json
{"object": "Transaction", "coin": "VAR" | "SKA1" | ..., "total": "<сумма>", "vout": "<n>", "vin": "<m>"}
```

Правила формата `total` (это **видимое пользователю число**, поэтому точность важна — в отличие от ширины сегмента):

- для VAR — десятичное значение с точностью до 8 знаков после запятой (например, `151.91101732`);
- для SKA{n} — десятичное значение с точностью до 18 знаков после запятой (например, `123.000000000000000001`), без экспоненциальной записи и без потери последних знаков.

**Объяснение (почему точность важна именно тут).** Сумма перевода — это то, что пользователь приходит проверить: «сколько именно ушло». Если в сегменте VAR-перевода на 151 целую часть «отрежется» дробная — пользователь решит, что это другая транзакция. Для SKA-переводов проблема ещё острее: типичные SKA-суммы могут иметь значащие цифры в последних знаках (например, малое дробное вознаграждение), и любое преобразование такого числа в формат с плавающей точкой потеряет именно эти значащие цифры. Поэтому SKA-значения переносятся от сервера до подсказки **в виде строки**, без промежуточного «округления через плавающую точку».

### 6.4. Цвет / визуальное различие монет

Сегменты обычных транзакций различаются цветом по принципу «VAR vs не-VAR»:

- **VAR-tx** — текущий цвет (без изменений);
- **Любая SKA{n}-tx** — единый альтернативный цвет, общий для всех SKA-типов.

Альтернативный цвет подбирается **близким по светлоте и насыщенности** к существующему VAR-цвету и отличается только тоном (hue). Цель — дать пользователю быстрый признак «этот сегмент — VAR или SKA» одним взглядом, **не перетягивая визуальный вес** у соседних сегментов. Тонкий SKA-сегмент рядом с широким VAR-сегментом не должен выглядеть «громче» только из-за цвета — это нарушило бы шкалу пропорций по ширине (§3.5, §6.2).

Различение конкретных SKA{n} между собой (SKA1 vs SKA2 vs …) — **вне области данного спека**. Конкретный SKA-тип читается из заголовка плитки (§2.2) и всплывающей подсказки (§6.3).

Требования к обоим пайплайнам рендеринга (HTTP-шаблон и JS-конструктор плиток):

- класс/маркер для SKA-сегмента добавляется в **обе** ветки одновременно (см. §8);
- тёмная и светлая темы должны давать сопоставимый визуальный вес VAR- и SKA-сегментов.

## 7. Плитка мемпула

Плитка мемпула наследует все правила §2–§6:

- заголовок-сумма (вместо `486 DCR`) — мультимонетный вертикальный список (см. §2.2);
- Rewards row — только PoW (VAR + SKA{n}) и PoS (VAR); сегмент `fund` удалён; Fees — мультимонетные;
- Votes / Tickets / Revocations — без логических изменений;
- Regular transactions row — мультимонетные, по той же шкале, что и в плитке блока.

Все правила точности и шкалы §3.5/§6.2 применяются и к мемпулу.

**Источник данных для мемпула** на странице:

- HTTP: `MempoolInfo.Trim()` + патч `Subsidy = HomeInfo.NBlockSubsidy` (как сегодня);
- WS: ответ `getmempooltrimmed` + тот же патч.

Поскольку `HomeInfo.NBlockSubsidy` теперь должен нести мультимонетную субсидию (см. [homepage-metrics/spec.md §4.2](../homepage-metrics/spec.md)), при апдейте бэкенд-структуры необходимо учесть **обе** точки патчинга (`explorerroutes.go` и `websockethandlers.go`), а также **дубликат расчёта в `pubsub/pubsubhub.go`** — см. [code-analysis/visualblocks/patterns.md §7](../../code-analysis/visualblocks/patterns.md).

## 8. Live-обновления

Двойной пайплайн обновлений сохраняется:

- **HTTP рендер** — первичный показ страницы шаблоном `visualblocks.tmpl`;
- **WebSocket `newblock`** — приходит **полный** `BlockInfo`; JS-контроллер строит новую плитку и вставляет в DOM, удаляя самую старую, чтобы держать в DOM ≤ 30 плиток (DOM-кэп, см. §1); параллельно пересчитывается, сколько из этих плиток помечено `visible` исходя из размера окна;
- **WebSocket `getmempooltrimmed`** — приходит `TrimmedMempoolInfo` + патченая субсидия; JS-контроллер заменяет первую плитку (мемпул).

Требования:

- HTML-структура плитки, генерируемая в JS, должна быть **идентична** структуре из шаблона (теги, классы, формат `title`-JSON, отсутствие `class="fund"`, отсутствие подписи `DCR`);
- любое новое поле в `TrimmedBlockInfo` / `BlockInfo` (например, мультимонетные суммы транзакций или per-coin субсидия) должно добавляться **на обоих путях одновременно** — шаблон и JS-контроллер;
- никакая мутация указателя `*BlockInfo`, возвращённого `GetExplorerBlock`, на UI-слое **не допускается** (это разделяемый memoized-указатель — см. [code-analysis/visualblocks/patterns.md §5](../../code-analysis/visualblocks/patterns.md)).

## 12. Вне области действия

- Маршрут `/visualblocks` остаётся включённым (страница не отключается).
- DOM-кэп «не более 30 плиток блоков плюс одна плитка мемпула» сохраняется как есть.
- Динамический расчёт видимых плиток по размеру окна (CSS-класс `visible`, пересчёт на загрузку и `resize`) сохраняется как есть — мультимонетные изменения его не затрагивают.
- Правило обрезания «толстого» блока сохраняется как есть — это поведение унаследовано от Decred-кодовой базы. Если в блоке оказывается **более 50 билетов** или **более 50 обычных транзакций**, соответствующая строка плитки отображает только **первые 30 сегментов**, а в конец строки добавляется индикатор «+N more» (где N — число обрезанных элементов). Порог одинаков для обеих строк (билеты и обычные транзакции) и применяется по их общему количеству, без разбивки по монетам — то есть мультимонетные правки этой механики не затрагивают: пятьдесят VAR-tx + ноль SKA-tx и тридцать VAR-tx + двадцать SKA-tx считаются одинаково и одинаково попадают под обрезание. Голоса (cap 5) и ревокации (cap 20 - голоса - билеты) под это правило не подпадают — у них собственные стейкинговые «потолки» (см. §4, §5). Обрезание применяется симметрично в обоих пайплайнах рендеринга (HTTP-шаблон и JS-контроллер, см. §1, §8): и при первоначальной загрузке страницы, и при живом обновлении плитки по WebSocket.
- Количество пустых «слотов» для голосов (5) и для билетов+ревокаций (20) на плитке сохраняется — это параметры стейкинга, к мультимонетной модели они отношения не имеют.
- Логика отбрасывания коинбейс-транзакции из строки обычных транзакций сохраняется и на сервере, и в JS.
- Кеш «последнего собранного блока» на стороне БД не меняется; правило не мутировать возвращаемые данные блока в обработчиках сохраняется.
- Порядок захвата внутренних блокировок при обновлении состояния страницы сохраняется (см. [code-analysis/visualblocks/patterns.md §6](../../code-analysis/visualblocks/patterns.md)).

## 13. Критерии приёмки

- [ ] На странице нигде не отображается подпись `DCR`.
- [ ] В каждой плитке (плитка мемпула + любое число плиток блоков, фактически видимое в окне; в DOM их до 30) сумма блока выводится **вертикальным мультимонетным списком** (VAR обязательно, SKA{n} — только при наличии).
- [ ] Сегмент строки наград «Project Fund» отсутствует — как при первоначальной загрузке страницы, так и при живых обновлениях плиток.
- [ ] В строке наград PoW отражает PoW-вознаграждения всех монет блока (VAR + эмитированные SKA{n}); всплывающая подсказка показывает разбивку по монетам.
- [ ] Всплывающая подсказка сегмента комиссий показывает разбивку по монетам; монеты с нулевой комиссией в разбивке не показываются.
- [ ] Голоса, билеты и ревокации остаются VAR-only; их всплывающая подсказка явно содержит `"coin": "VAR"`.
- [ ] Каждый сегмент обычной транзакции во всплывающей подсказке имеет поле `coin` со значением `VAR` или `SKA{n}`.
- [ ] Сегменты VAR-tx и SKA-tx визуально отличаются цветом (один общий цвет для всех SKA{n}); тонкий SKA-сегмент не выглядит «громче» широкого VAR-сегмента только из-за цвета — визуальный вес сопоставим, шкала пропорций по ширине не нарушена.
- [ ] Числовые суммы, видимые пользователю (в заголовке плитки и во всплывающих подсказках), не теряют значащих цифр для SKA-значений — даже если внутреннее представление суммы имеет 18 знаков после запятой.
- [ ] Сегмент крупной SKA-транзакции не «съедает» строку обычных транзакций из-за разницы в шкале единиц VAR vs SKA — сегменты разных монет визуально сопоставимы по доле в собственной монете, а не по сырому числу атомов.
- [ ] Пустые слоты голосов (5) и билетов/ревокаций (20) на плитку сохранены.
- [ ] Живое обновление плитки при приходе нового блока даёт **структурно идентичную** разметку с тем, что отдаёт серверный шаблон при первой загрузке (теги, классы, формат JSON в подсказках).
- [ ] Живое обновление плитки мемпула даёт разметку, идентичную серверной.
- [ ] Treasury / Project Fund нигде не отображаются ни в одной плитке.
- [ ] Тёмная тема корректно отрабатывает на изменённой странице.
- [ ] Значения субсидии за блок, поступающие в плитку мемпула, согласованы между основным каналом обновлений страницы и параллельным pub/sub-каналом (см. §7).
