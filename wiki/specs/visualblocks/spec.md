# Visual Blocks — поддержка мультимонетной модели Monetarium (requirements)

## 0. Контекст

Страница `/visualblocks` отображает **последние блоки и текущий мемпул в виде «плиток»**, где элементы блока (голоса, билеты, обычные транзакции) визуализированы горизонтальными цветными строками. Раздел унаследован от Decred-кодовой базы и в текущем виде поддерживает только одну монету (DCR), модель Project Fund (Decred Treasury) и одну шкалу субсидии на блок.

В Monetarium действуют другие инварианты (см. [CLAUDE.md](../../../CLAUDE.md), [core/product.md](../../core/product.md), [core/pages.md](../../core/pages.md)):

- Поддерживается несколько монет: **VAR** и до 255 типов **SKA{n}** (SKA1, SKA2, …).
- **Один блок может содержать транзакции нескольких монет**, однако **одна транзакция всегда односоставна по монете**.
- **Комиссия транзакции уплачивается в монете самой транзакции** (VAR-tx → fee в VAR, SKA{n}-tx → fee в SKA{n}).
- **Treasury (Project Fund) отсутствует** — соответствующая шкала в Decred-схеме на Monetarium неприменима.
- Staking-механика остаётся **VAR-only** (голосование, билеты, ревокации существуют только в VAR).
- **Точность чисел:** VAR — 8 знаков (помещается в `float64`); SKA — 18 знаков (**не помещается в `float64`**). См. §0.2 — после правок из созвона страница в v1 **не отображает суммы в монетах**, поэтому ограничение точности на этой странице в v1 не задействовано.

Данный документ описывает, что нужно **создать / обновить / удалить** на странице `/visualblocks`, чтобы привести её в соответствие с мультимонетной моделью Monetarium.

### 0.1. Происхождение требований и разрешённые конфликты

Этот документ — **результат мержа** двух источников:

1. исходные мультимонетные требования к странице;
2. требования, зафиксированные на созвоне с заказчиком (бывший черновик `call-spec.md`, теперь поглощён этим документом).

Конфликты между двумя источниками разрешены **в пользу требований созвона** (требования заказчика приоритетны). Разрешения:

| Конфликт                                | Старое требование                                | Решение (созвон, принято)                                                                 |
| --------------------------------------- | ------------------------------------------------ | ----------------------------------------------------------------------------------------- |
| Центр заголовка плитки                  | Вертикальный список сумм по монетам (VAR + SKA{n}) | **Размер блока** (байты / КБ) + опционально «% от максимального размера блока» (см. §2.2)  |
| Строка наград (Rewards row)             | Сохранить и переделать под мультимонетность       | **Удалить строку наград полностью** (см. §3)                                               |
| Строка обычных транзакций               | Сохранить, делать сегменты мультимонетными         | **Заменить тремя полосами индикатора** TOTAL / VAR / SKA{n} (см. §6)                       |

Дополнительно в этот документ внесены **решения из обсуждения PR #262** (см. §4, §6.4 — пометки «Решение PR»).

### 0.2. Следствие для точности чисел на этой странице

После правок созвона страница `/visualblocks` в v1 **не отображает суммы в монетах ни в одном месте**:

- заголовок плитки показывает **размер блока в байтах** (целое, не сумма в монете);
- всплывающие подсказки полос индикатора показывают **количество транзакций** (целое);
- заполнение полос индикатора — это **отношение размеров в байтах** (байты / байты).

Поэтому общее ограничение точности VAR (8 знаков) / SKA (18 знаков, строкой/`big.Int`, без `float64`) из [core/constraints.md](../../core/constraints.md) на этой странице в v1 **не задействовано**. Ограничение остаётся в силе на уровне проекта и снова станет актуальным, если в следующих итерациях на плитку вернут отображение сумм.

## 1. Структура страницы (общий вид)

Структура страницы сохраняется:

- слева — плитка **Mempool** (текущее состояние мемпула);
- справа от неё — последовательность плиток **последних блоков** по убыванию высоты.

Сколько плиток блоков пользователь видит — **зависит от размера окна**: страница подстраивается под доступную ширину и высоту и показывает столько плиток, сколько помещается без обрезания и переноса; то же пересчитывается при изменении размера окна. На широком мониторе плиток видно больше, на узком — меньше; самая свежая всегда левее самой старой. При приходе нового блока самая свежая плитка появляется слева сразу после плитки мемпула, а самая старая из видимых уходит с экрана.

Это поведение **сохраняется как есть** — мультимонетные правки его не затрагивают.

> Для разработчиков: текущая реализация держит небольшой DOM-кэш плиток (фиксированный потолок на стороне сервера) и переключает видимость каждой плитки CSS-классом на клиенте (пересчёт при загрузке и на `resize`). При мультимонетных правках важно не поменять эти инварианты случайно — не увеличивать/уменьшать серверный потолок, не ломать пересчёт видимости и не оставлять «осиротевшие» плитки в DOM при приходе нового блока.

Внутри каждой плитки после правок остаются **три строки** (строка наград удалена, см. §3; строка обычных транзакций заменена полосами индикатора, см. §6):

1. **Votes row** — голосования (VAR-only, три состояния — см. §4).
2. **Tickets row** — билеты + ревокации (VAR-only, см. §5).
3. **Полосы индикатора блока** — TOTAL / VAR / SKA{n} (заменяют строку обычных транзакций, см. §6).

После удаления строки наград **строка голосов идёт сразу за заголовком плитки**.

**Терминология** (используется по всему документу):

- **Плитка** (`block tile` / `mempool tile`) — большой элемент сетки: одна за мемпул, до 30 за последние блоки. Содержит заголовок и строки.
- **Строка** (`row`) — горизонтальная строка внутри плитки (`block-votes` / `block-tickets`).
- **Сегмент** — отдельный элемент внутри строки билетов/голосов: отдельный голос / билет / ревокация.
- **Полоса индикатора** — компонент `indicator-fill` (тот же, что на главной странице), заполняемая шкала; на плитке их несколько: TOTAL, VAR, по одной на каждую активную SKA-монету.

> Шаблон и динамический рендер по WebSocket должны выдавать **структурно идентичный DOM** (классы, `data-*`-хуки, формат `title`). Это критично из-за наличия двух пайплайнов рендеринга (HTTP-шаблон и JS-контроллер), которые мутируют один и тот же DOM — см. [code-analysis/visualblocks/patterns.md](../../code-analysis/visualblocks/patterns.md).

## 2. Заголовок плитки (Block info / Mempool info)

### 2.1. Кликабельный заголовок плитки

Без изменений:

- для плитки мемпула — «Mempool» с ссылкой `/mempool`;
- для плитки блока — высота блока с ссылкой `/block/{height}`.

### 2.2. Центр заголовка — размер блока вместо суммы в DCR

**Текущее поведение:**
Центральная часть заголовка показывает общую сумму блока с единицей «DCR» (например, «486 DCR»). Значение — `block.TotalSent` в Decred.

**Требуемое поведение:**

- Жёстко зашитая подпись `DCR` и сумма в DCR **должны быть удалены** во всех местах:
  - шаблон [`cmd/dcrdata/views/visualblocks.tmpl`](../../../cmd/dcrdata/views/visualblocks.tmpl) (`<span class="unit"> DCR</span>` для плитки мемпула и каждой плитки блока);
  - JS-контроллер [`cmd/dcrdata/public/js/controllers/visualBlocks_controller.js`](../../../cmd/dcrdata/public/js/controllers/visualBlocks_controller.js) (`makeMempoolBlock`, `_makeBlockNode`).
- Вместо суммы отображается **размер блока** (в байтах / КБ).
- Если в заголовок помещается — дополнительно показывать **% от максимального размера блока**.
- Если не помещается — минимум: показать только размер блока.
- Для плитки мемпула в центре заголовка показывается **текущий суммарный размер мемпула** (байты / КБ), по той же логике (+ опционально % от максимального размера блока).
- Максимальный размер блока берётся из chain spec и **передаётся сервером** (одно значение `maxBlockSize`, общее для страницы).

Остальные элементы заголовка не изменяются (см. §2.1, §2.3).

**Данные:** `Block.Size` (uint32, байты) — `db/dbtypes/types.go`; `BlockBasic.FormattedBytes` (string) — `explorer/types/explorertypes.go`. Идея «вертикального списка сумм по монетам» в заголовке **отменена** (конфликт разрешён в пользу созвона, см. §0.1) — серверная сводка сумм блока по монетам для заголовка **не требуется**.

### 2.3. Возраст блока (Age)

Без изменений: относительное время через `data-time-target="age"`.

## 3. Rewards row (строка наград) — удалена

**Текущее поведение:**
ROW 1 (`block-rewards`) — зелёная полоса, показывает пропорцию между PoW-наградой майнеру, PoS-наградой стейкерам, Project Fund и комиссиями (сегменты разной яркости зелёного).

**Требуемое поведение: строку наград удалить полностью.**

**Причина (созвон).** Полоса показывает соотношение PoW-награды и PoS-награды. В Monetarium оно всегда фиксировано (50/50), поэтому полоса не несёт информации. Project Fund (Decred Treasury) в Monetarium отсутствует. После удаления строки наград **строка голосов (см. §4) идёт сразу за заголовком плитки**.

**Объём удаления:**

- сегменты `pow`, `pos`, `fees`, `fund` и контейнер строки `block-rewards` — из шаблона [`visualblocks.tmpl`](../../../cmd/dcrdata/views/visualblocks.tmpl) (обе позиции: плитка мемпула и плитка блока);
- те же ветки из JS-контроллера (`makeMempoolBlock`, `_makeBlockNode`), включая чтение `subsidy.pow / subsidy.pos / subsidy.fees / subsidy.developer || subsidy.dev`;
- сопутствующие CSS-правила (`.pow`, `.pos`, `.fees`, `.fund` и стиль контейнера строки) в [`public/scss/visualblocks.scss`](../../../cmd/dcrdata/public/scss/visualblocks.scss) допускается удалить или оставить «осиротевшими»; визуально они нигде не должны проявляться.

**Следствия для остальной части документа:**

- Поскольку строка наград удалена, **мультимонетная разбивка PoW / PoS / комиссий на этой странице не реализуется** (она имела смысл только в подсказках сегментов строки наград).
- Субсидия больше **не отображается** на плитке. Соответственно патч `Subsidy` для плитки мемпула и его синхронизация с дублирующим расчётом в `pubsub/pubsubhub.go` для этой страницы **больше не требуются** (см. §7).
- Серверные сводки «PoW-награды по монетам» и «комиссии по монетам» для этой страницы **не нужны** (см. §10).

## 4. Votes row (строка голосов) — три состояния

Голосование в Monetarium — VAR-only, структура голосов наследуется из Decred-схемы. После удаления строки наград (§3) строка голосов идёт **сразу за заголовком плитки**.

**Текущее поведение:**
5 слотов; голос «за» = синий, голос «против» = красный, пустой слот = нет цвета. Различие двух состояний — через `voteValid` (true/false).

**Требуемое поведение (созвон):**
На каждый блок случайно выбираются ровно **5 билетов**. Кошелёк-владелец может ответить «да», ответить «нет», либо оказаться офлайн и не проголосовать вовсе. **Три состояния** элемента голоса:

| Статус                           | Цвет        | Описание                                                                  |
| -------------------------------- | ----------- | ------------------------------------------------------------------------- |
| Voted YES — проголосовал «за»    | **Синий**   | Согласен с блоком; возвращается стоимость билета + награда за голос         |
| Voted NO — проголосовал «против» | **Красный** | Не согласен с блоком (редко: сбой или атака)                               |
| Did not vote — не проголосовал   | **Серый**   | Кошелёк был офлайн; возвращается **только** стоимость билета, награды нет   |

Дополнительные правила:

- «Пустые слоты» (для выравнивания до 5, если проголосовало меньше 5 билетов) — оставить прозрачными (текущее поведение). Константа `5` остаётся захардкоженной как stake-параметр.
- Ссылка голоса на `/tx/<hash>` сохраняется.

**Изменения по данным и разметке:**

- текущий код различает два состояния через `voteValid` (true/false); третье состояние «did not vote» требует **дополнительного поля** в данных голоса (флаг «голос подан / не подан», отдельный от `voteValid`). Это поле добавляется **на обоих транспортах** (HTTP-шаблон и WebSocket — см. §8).
- из `title`-tooltip убрать любые скрытые упоминания DCR;
- `title` голоса должен нести **три состояния** и явную монету, например: `{"object": "Vote", "coin": "VAR", "voted": "<true|false>", "voteValid": "<true|false>"}` (поле `voteValid` имеет смысл только при `voted=true`).

> **Решение PR #262 (обсуждение строки билетов, см. §5).** В ветке обсуждения было подтверждено код-анализом, что строка билетов и ревокаций сохраняется как есть; правки голосов (три состояния) — отдельны и совместимы.

## 5. Tickets row (строка билетов и ревокаций) — без логических изменений

Билеты и ревокации в Monetarium — VAR-only. Жёлтые купленные билеты + оранжевые ревокации + прозрачные пустые слоты — **поведение сохраняется как есть**.

**Изменения по разметке:**

- в `title`-tooltip билета явно добавить `"coin": "VAR"`:
  `{"object": "Ticket", "coin": "VAR", "total": "<VAR-amount>", "vout": "...", "vin": "..."}`;
- аналогично для ревокаций (`{"object": "Revocation", "coin": "VAR", ...}`);
- `total` отображается в VAR с 8 знаками точности (через `float64` допустимо — VAR помещается в `float64`).

### 5.1. Семантика пустых слотов и числа 20 (уточнено по коду в PR #262)

В обсуждении PR #262 было разобрано по существующему коду (шаблон `visualblocks.tmpl`, принято заказчиком):

- **Билеты и ревокации рендерятся безусловно в одном общем ряду** — лимита «20 слотов под билеты» в коде нет.
- Число **20** — это **захардкоженный литерал**, используемый **только** для расчёта количества прозрачных слотов-заполнителей по формуле:

  `пустых_слотов = max(0, 20 − (ticketsCount + revocationsCount))`

  (отрицательная длина обрезается до нуля).
- 20 — это **общий пул «билеты + ревокации»** для добивки пустыми слотами, **а не** «20 слотов под билеты» и **не** «20 − votes − tickets − revocations» (голоса — отдельная строка со своим потолком 5, в формулу строки билетов **не входят**).
- Если `tickets + revocations > 20` → пустых слотов **0**, но все билеты и все ревокации всё равно нарисованы; **ряд просто становится длиннее 20** (например, 20 билетов + 2 ревокации = 22 видимых элемента). Explorer это никак не ограничивает.
- Консенсусный лимит `MaxFreshStakePerBlock = 20` ограничивает только покупку новых билетов (SSTx), не связан с ревокациями, и **шаблон эту константу не использует** — в коде стоит «голая» цифра 20 (зафиксировано в [code-analysis/visualblocks/flow.compact.md](../../code-analysis/visualblocks/flow.compact.md)).

Поведение сохраняется как есть; формула пустых слотов в этом документе приведена в исправленном виде именно по результату код-анализа PR.

## 6. Полосы индикатора блока (заменяют строку обычных транзакций)

**Текущее поведение:**
ROW 4 (`block-transactions`) — серая полоса; каждая обычная транзакция — отдельный серый (`#c4cbd2`) сегмент, ширина пропорциональна сумме транзакции.

**Требуемое поведение (созвон):**
Серую строку транзакций **удалить полностью** (оба пайплайна: шаблон и JS-контроллер, включая per-tx сегменты и связанный CSS) и заменить **полосами индикатора** из компонента `indicator-fill` — **того же компонента, что на главной странице**:

- **TOTAL bar** — одна;
- **VAR bar** — одна;
- **SKA{n} bar** — по одной на каждую **активную** SKA-монету.

Идея «мультимонетных per-tx сегментов» (разный цвет VAR vs SKA, ширина «в рамках своей монеты», `coin` в подсказке каждой транзакции) **отменена** вместе с самой строкой транзакций (конфликт разрешён в пользу созвона, см. §0.1). Соответственно «тихий риск» с SKA-сегментом, съедающим строку, более неактуален — агрегированные полосы по байтам его исключают.

### 6.1. Алгоритм квот (как на главной)

- `VAR quota = 10% от maxBlockSize`.
- `SKA pool = 90% от maxBlockSize`, делится **поровну** между всеми **активными** SKA-монетами: `SKA{n} quota = SKA pool / activeSkaCount`.
- `maxBlockSize` — то же серверное значение, что в §2.2.

### 6.2. Плитка мемпула — заполнение полос

Идентично индикатору на главной странице:

- **TOTAL bar** — общее заполнение мемпула vs. `maxBlockSize`.
- **VAR bar** — заполнение VAR-транзакциями vs. VAR-квота.
- **SKA{n} bar** — заполнение SKA{n}-транзакциями vs. SKA{n}-квота.

Источник данных: `TrimmedMempoolInfo.CoinFills` + `CoinStats` — **уже вычислены сервером** и передаются по WebSocket (`getmempooltrimmed`). **Изменений в серверной части для полос мемпула не требуется.**

### 6.3. Плитка намайненного блока — заполнение полос

Те же три типа полос, но **по фактическим данным блока** (всё в байтах):

- **TOTAL bar** — фактический размер блока / `maxBlockSize`.
- **VAR bar** — суммарный размер VAR-транзакций блока / VAR-квота.
- **SKA{n} bar** — суммарный размер SKA{n}-транзакций блока / SKA{n}-квота.

Для этого серверная модель блока, отдаваемая в шаблон и в WebSocket, должна нести: фактический размер блока, **суммарные размеры обычных транзакций по монетам** (VAR и каждая SKA{n}) и число активных SKA-монет для расчёта квот (см. §10). Коинбейс из подсчёта обычных транзакций исключается (как сегодня, на сервере и в JS).

### 6.4. Всплывающие подсказки полос индикатора

Визуальный компонент (`indicator-fill`) один и тот же для обоих типов плиток. Подсказки на `/visualblocks` показывают **количество транзакций** (в штуках), а не процент занятого места (как на главной).

**Плитка мемпула — количество + «% от мемпула».**
Мемпул — живой срез: знаменатель (количество транзакций в мемпуле) доступен прямо сейчас, той же размерности (штуки), что и числитель.

| Полоса | Содержимое подсказки                         |
| ------ | -------------------------------------------- |
| TOTAL  | «N транзакций всего — X% от мемпула»          |
| VAR    | «N VAR-транзакций — X% от VAR-мемпула»        |
| SKA{n} | «N SKA{n}-транзакций — X% от SKA{n}-мемпула»  |

**Плитка намайненного блока — только количество транзакций (v1).**

> **Решение PR #262 (обсуждение, принято заказчиком).** Для намайненного блока корректного процента в подсказке в v1 **не будет**. Причина: подсказка показывает количество транзакций (штуки), а единственный доступный без исторических данных знаменатель `maxBlockSize` выражен в **байтах** — это **несовместимые единицы** (штуки vs байты), деление бессмысленно. Состояние мемпула на момент майнинга в БД не хранится, поэтому «% от мемпула» для блока **не восстановим**. Решено: в v1 в подсказке плиток намайненных блоков показывать **только количество транзакций**, без процента. Реализацию процента (количество tx относительно мемпула на момент создания блока) **отложить до следующей итерации**.

| Полоса | Содержимое подсказки (v1) |
| ------ | ------------------------- |
| TOTAL  | «N транзакций всего»      |
| VAR    | «N VAR-транзакций»        |
| SKA{n} | «N SKA{n}-транзакций»     |

### 6.5. Отложено до следующей итерации — исторический snapshot мемпула

Чтобы в подсказках плиток намайненных блоков показывать процент относительно мемпула на момент майнинга, потребуется историческое хранение snapshot мемпула:

- новая таблица БД (схема + миграция), включая per-coin breakdown;
- новая реализация `MempoolDataSaver`, пишущая в БД;
- синхронизация порядка вызовов `BlockHandler` ↔ `StoreBlock()` (конкурентны в одной группе — нужно согласовать `block_id`);
- обработка реорганизаций цепи (orphan → «висячий» snapshot: `ON DELETE CASCADE` на `block_id` либо явное удаление в Go);
- ретроспективы нет: snapshot доступен только для блоков начиная с момента развёртывания.

**Оценка:** 2–4 дня; основная сложность — синхронизация порядка вызовов и реорганизации (в `MempoolInfo.CoinStats` сами данные уже есть). В обсуждении PR #262 сложность этого варианта была подтверждена заказчиком; **в v1 не реализуется**.

## 7. Плитка мемпула

Плитка мемпула наследует все правила §2–§6:

- центр заголовка — **текущий размер мемпула** (вместо `486 DCR`), + опционально % от `maxBlockSize` (см. §2.2);
- **строки наград нет** (удалена, см. §3);
- Votes / Tickets / Revocations — как в плитке блока (Votes — три состояния, §4; Tickets — без логических изменений, §5);
- полосы индикатора TOTAL / VAR / SKA{n} — по данным мемпула (см. §6.2), компонент тот же, что на главной.

**Источник данных для мемпула:**

- HTTP: `MempoolInfo.Trim()` (как сегодня);
- WS: ответ `getmempooltrimmed`.

Поскольку строка наград удалена, **патч `Subsidy = HomeInfo.NBlockSubsidy` для этой страницы больше не нужен**, и связанная с ним синхронизация субсидии с дублирующим расчётом в `pubsub/pubsubhub.go` (бывший §7 исходного спека) на `/visualblocks` **не требуется**. Полосы индикатора мемпула питаются из `TrimmedMempoolInfo.CoinFills` + `CoinStats`, которые сервер уже вычисляет (см. §6.2) — дополнительных серверных правок для мемпульной плитки не нужно.

## 8. Live-обновления

Двойной пайплайн обновлений сохраняется:

- **HTTP рендер** — первичный показ страницы шаблоном `visualblocks.tmpl`;
- **WebSocket `newblock`** — приходит **полный** `BlockInfo`; JS-контроллер строит новую плитку и вставляет в DOM, удаляя самую старую, чтобы держать ≤ 30 плиток (DOM-кэп, §1); параллельно пересчитывается видимость по размеру окна;
- **WebSocket `getmempooltrimmed`** — приходит `TrimmedMempoolInfo`; JS-контроллер заменяет первую плитку (мемпул).

Требования:

- HTML-структура плитки, генерируемая в JS, должна быть **идентична** структуре из шаблона: теги, классы, формат `title`-JSON; **отсутствие `class="fund"` и всей строки наград**, **отсутствие подписи `DCR`**, **отсутствие per-tx сегментов строки транзакций**, наличие полос индикатора (TOTAL / VAR / SKA{n}) и их подсказок;
- любое новое поле (размер блока, `maxBlockSize`, per-coin размеры транзакций, число активных SKA, флаг «голос подан») должно добавляться **на обоих путях одновременно** — шаблон и JS-контроллер;
- никакая мутация указателя `*BlockInfo`, возвращённого `GetExplorerBlock`, на UI-слое **не допускается** (разделяемый memoized-указатель — см. [code-analysis/visualblocks/patterns.md §5](../../code-analysis/visualblocks/patterns.md)).

## 12. Вне области действия

- Маршрут `/visualblocks` остаётся включённым (страница не отключается).
- DOM-кэп «не более 30 плиток блоков плюс одна плитка мемпула» сохраняется как есть.
- Динамический расчёт видимых плиток по размеру окна (CSS-класс `visible`, пересчёт на загрузку и `resize`) сохраняется как есть.
- **Правило обрезания «толстого» блока сохраняется только для строки билетов** (строка обычных транзакций удалена — см. §6, — поэтому её прежнее правило обрезания `+N more` отпадает вместе с per-tx сегментами). Для строки билетов правило сохраняется как унаследованное: если в блоке **более 50 билетов**, строка отображает только **первые 30** сегментов, а в конец добавляется индикатор «+N more». Голоса (cap 5) и ревокации под это правило не подпадают. Обрезание применяется симметрично в обоих пайплайнах рендеринга (HTTP-шаблон и JS-контроллер): и при первой загрузке, и при живом обновлении по WebSocket.
- Семантика пустых слотов сохраняется как есть: голоса — потолок 5; пустые слоты строки билетов — `max(0, 20 − (билеты + ревокации))`. Число 20 — захардкоженная база отрисовки прозрачных слотов, смешивающая в одном пуле билеты и ревокации, **а не** консенсусный лимит и **не** «20 слотов под билеты» (см. §5.1; уточнено код-анализом в PR #262). Эти числа — stake-параметры, к мультимонетной модели отношения не имеют.
- Логика отбрасывания коинбейс-транзакции из подсчёта обычных транзакций сохраняется и на сервере, и в JS (учитывается в счётчиках полос индикатора, §6.3).
- Кеш «последнего собранного блока» на стороне БД не меняется; правило не мутировать возвращаемые данные блока в обработчиках сохраняется.
- Порядок захвата внутренних блокировок при обновлении состояния страницы сохраняется (см. [code-analysis/visualblocks/patterns.md §6](../../code-analysis/visualblocks/patterns.md)).
- Исторический snapshot мемпула для процента в подсказках намайненных блоков — **отложен** (§6.5), в v1 не реализуется.

## 13. Критерии приёмки

- [ ] На странице нигде не отображается подпись `DCR` и сумма в DCR.
- [ ] В каждой плитке (мемпул + плитки блоков) центр заголовка показывает **размер блока** (для мемпула — текущий размер мемпула); при наличии места дополнительно показан «% от максимального размера блока».
- [ ] **Строка наград полностью отсутствует** — как при первой загрузке, так и при живых обновлениях; строка голосов идёт сразу за заголовком плитки.
- [ ] Сегмент «Project Fund» / Treasury нигде не отображается ни в одной плитке.
- [ ] Строка голосов имеет **три состояния**: синий (YES), красный (NO), серый (did not vote); пустые слоты до 5 сохранены и прозрачны; `title` голоса содержит `"coin": "VAR"` и различает «голос подан / не подан».
- [ ] Строка билетов/ревокаций сохранена как есть; `title` билета и ревокации содержит `"coin": "VAR"`; пустые слоты считаются как `max(0, 20 − (билеты + ревокации))`.
- [ ] Серая строка обычных транзакций (per-tx сегменты) **заменена** полосами индикатора TOTAL / VAR / SKA{n} на обоих типах плиток; компонент идентичен индикатору главной страницы.
- [ ] Квоты рассчитаны верно: VAR = 10% `maxBlockSize`; каждая SKA{n} = 90% `maxBlockSize` / число активных SKA-монет.
- [ ] Подсказки полос **плитки мемпула** показывают количество транзакций и «% от мемпула» (TOTAL/VAR/SKA{n}).
- [ ] Подсказки полос **плитки намайненного блока** в v1 показывают **только количество транзакций** (без процента).
- [ ] Живое обновление плитки при приходе нового блока даёт **структурно идентичную** разметку с серверным шаблоном (теги, классы, формат JSON в подсказках, отсутствие строки наград и DCR, наличие полос индикатора).
- [ ] Живое обновление плитки мемпула даёт разметку, идентичную серверной.
- [ ] Правило обрезания «+N more» работает для строки билетов при > 50 билетов (обе ветки рендера); строка обычных транзакций отсутствует, её прежнее правило обрезания неприменимо.
- [ ] Тёмная тема корректно отрабатывает на изменённой странице (полосы индикатора, цвета трёх состояний голосов).
